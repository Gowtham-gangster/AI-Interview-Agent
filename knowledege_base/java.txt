PolymorphismPolymorphism is the ability of a single entity (method, object, or reference) to take **multiple forms**.In Java it appears in two ways:| Type | Description | How it is achieved ||------|-------------|--------------------|| **Compile‑time (static) polymorphism** | The compiler determines which method to invoke. | **Method overloading** – same method name with different parameter lists. || **Run‑time (dynamic) polymorphism** | The method to execute is decided at run time based on the actual object type. | **Method overriding** – subclass provides its own implementation of a method declared in a superclass. |Thus, polymorphism enables code that works on a **supertype reference** to operate on many different **subtype objects**, promoting flexibility and extensibility.### 2. Method OverloadingMethod overloading is a form of **compile‑time polymorphism** where a class declares **multiple methods with the same name** but **different parameter lists** (different number, type, or order of parameters).- The compiler determines which method to invoke based on the **static types of the arguments** at the call site.- Return type may differ, but it is **not** sufficient alone to distinguish overloaded methods.```javaclass MathUtil {&#x20;   int add(int a, int b)                { return a + b; }&#x20;   double add(double a, double b)      { return a + b; }&#x20;   int add(int a, int b, int c)         { return a + b + c; }}```Calling `add(2,3)` invokes the first method, while `add(2.0,3.0)` invokes the second.## Key Concepts* **Polymorphism** – one interface, many implementations (compile‑time via overloading, run‑time via overriding).* **Method Overriding** – subclass provides its own implementation of a superclass method; enables dynamic dispatch.* **Method Overloading** – same method name, different signatures; resolved at compile time.## Evaluation Points* Define polymorphism and list its two types.* Explain method overriding with rules and a short example.* Explain method overloading, how the compiler chooses the method, and give an example.## Common Mistakes* Confusing overloading with overriding (different signatures vs. identical signatures).* Saying return type alone can overload a method.* Forgetting that overriding requires the same method signature and obeys access‑modifier rules.## Follow‑up Questions* How does Java implement dynamic method dispatch?* What is the role of `@Override` annotation?* Can a static method be overridden? Fail‑Fast Iterator*Definition*A **Fail‑Fast iterator** checks whether the underlying collection has been structurally modified after the iterator was created. If a modification is detected (except through the iterator’s own `remove()`/`add()` methods), it throws a `java.util.ConcurrentModificationException`.*How it works*- The collection maintains a **modCount** field that is incremented on every structural change.- The iterator captures the current `modCount` when it is created.- On each call to `next()` or `hasNext()`, the iterator compares the saved count with the collection’s current `modCount`.- Mismatch → `ConcurrentModificationException`.*When it is used*All standard collection implementations in the **java.util** package (e.g., `ArrayList`, `HashSet`, `HashMap`, `LinkedList`) provide Fail‑Fast iterators.*Example (HashMap – Fail‑Fast)*```javaimport java.util.HashMap;import java.util.Iterator;import java.util.Map;public class FailFastDemo {&#x20;   public static void main(String\[] args) {&#x20;       Map<Integer, String> map = new HashMap<>();&#x20;       map.put(1, "one");&#x20;       map.put(2, "two");&#x20;       map.put(3, "three");&#x20;       Iterator<Map.Entry<Integer, String>> it = map.entrySet().iterator();&#x20;       while (it.hasNext()) {&#x20;           Map.Entry<Integer, String> e = it.next();&#x20;           if (e.getKey() == 1) {&#x20;               // structural change outside the iterator → exception&#x20;               map.remove(1);&#x20;           }&#x20;       }&#x20;   }}```**Output**```Exception in thread "main" java.util.ConcurrentModificationException&#x20;       at java.base/java.util.HashMap$HashIterator.checkForComodification(HashMap.java:1063)&#x20;       ...```---### 2. It achieves this by iterating over a **snapshot** (a copy) of the collection taken at the moment the iterator is created.*How it works*- The iterator works on a separate copy of the data (often a `java.util.concurrent` collection creates this copy internally).- Modifications to the original collection do not affect the iterator’s snapshot, and vice‑versa.*When it is used*All collections in the **java.util.concurrent** package (e.g., `ConcurrentHashMap`, `CopyOnWriteArrayList`, `CopyOnWriteArraySet`) provide Fail‑Safe iterators.*Example (ConcurrentHashMap – Fail‑Safe)*```javaimport java.util.Iterator;import java.util.Map;import java.util.concurrent.ConcurrentHashMap;public class FailSafeDemo {&#x20; This eliminates the need for manual `delete`/`free` calls that are required in languages like C/C++. || **Prevents memory‑leak bugs** | In large, long‑running applications (servers, desktop apps) forgetting to release objects can eventually exhaust the heap and cause `OutOfMemoryError`. GC automatically reclaims such “orphaned” objects. || **Simplifies development** | Developers can focus on business logic rather than low‑level memory management. Types of GC in the HotSpot JVM| GC Type | Generation(s) affected | When it runs | Typical pause time | What happens to objects ||---------|------------------------|--------------|--------------------|--------------------------|| **Minor GC** (young‑gen GC) | **Young generation** only (Eden + two Survivor spaces) | Frequently, after the Eden space fills up (allocation‑rate driven) | Usually **few milliseconds** (depends on heap size and collector) | Objects that are still reachable after the collection are **promoted** to one of the Survivor spaces; those that survive several minor GCs are eventually **promoted to the old generation**. || **Major GC** (old‑gen GC) | **Old generation** (sometimes also the young generation) | When the old generation reaches a certain occupancy threshold or when a promotion from young fails | Longer than minor GC – **tens to hundreds of ms** for Parallel/Serial collectors; can be higher for compacting collectors | Reclaims memory occupied by long‑living objects that are no longer reachable. In many collectors (e.g., Parallel, Serial) a major GC also **cleans the young generation**, but the focus is on the old gen. || **Full GC** (global GC) | **All generations** (young + old) and **Metaspace/PermGen** (if present) | Triggered when the JVM cannot allocate memory in either generation, when System.gc() is called, or when the collector decides it is more efficient to do a full collection | The **most expensive pause**, often **hundreds of ms** or even seconds for very large heaps. | Performs a complete **mark‑compact** (or mark‑sweep) across the entire heap, compacts live objects to eliminate fragmentation, and may also collect class metadata. |> \*\*Note:\*\* The exact terminology varies with the collector. In the Parallel and Serial collectors “major GC” and “full GC” are synonymous. In the G1 collector you will see “young GC” (minor) and “mixed GC” (partial old‑gen) and “Full GC” (full‑heap stop‑the‑world).  ### 2. failures manifest as `SQLException` (wrong URL, bad credentials, network issues). || Create statement | Provides a handle to send SQL; using the wrong type (`Statement` instead of `PreparedStatement`) can cause SQL injection and poorer performance. || Execute query | The actual work; a syntax error in SQL throws `SQLException`. || Process results | Must respect column types; calling the wrong getter (e.g., `getInt` on a VARCHAR column) may cause data conversion errors. || Close resources | Prevents connection leaks which eventually exhaust the connection pool or the DB’s allowed sessions. |### Alternatives for production‑grade code| Alternative | Benefit ||-------------|---------|| **`DataSource` + connection pool** (e.g., HikariCP, Apache DBCP) | Reuses physical connections, reduces latency, and provides built‑in pool management. || **JNDI lookup** (`InitialContext.lookup("java:comp/env/jdbc/MyDS")`) | Allows the container (Tomcat, JBoss, etc.) to manage the pool. || **ORM frameworks** (Hibernate, JPA) | Hide most of the boiler‑plate JDBC steps; still rely on the same underlying steps. || **Spring’s `JdbcTemplate`** | Simplifies exception handling and resource cleanup. |## Key Concepts* **JDBC driver** – translates JDBC calls to the database’s native protocol.* **`DriverManager`** – service that selects the appropriate driver based on the JDBC URL.* **`Connection`** – logical session, source of statements, and transaction manager.* **`Statement` / `PreparedStatement` / `CallableStatement`** – objects that send SQL to the DB.* **`ResultSet`** – cursor‑based view of query results.* **Resource cleanup** – essential to avoid leaks; use try‑with‑resources or finally blocks.## Evaluation Points* List the eight (or seven, if driver loading is omitted) steps in the correct order.* Provide a concise, compilable code snippet that follows the steps.* Mention the optional driver‑loading step and why it is usually unnecessary with modern drivers.* Explain the importance of closing resources and the advantage of try‑with‑resources.## Common Mistakes* Forgetting to close the `Connection` (or using `conn.close()` inside the loop).* Calling `Statement.executeQuery()` for an `INSERT`/`UPDATE` – use `executeUpdate()`.* Using `Class.forName` with an incorrect driver class name, causing `ClassNotFoundException`.* Not handling `SQLException` properly, leading to masked errors.## Follow‑up Questions* How does a connection pool improve performance and what are the key pool configuration parameters?* What is the difference between `Statement`, `PreparedStatement`, and `CallableStatement`?* How can you retrieve database metadata (tables, columns) using JDBC?* How would you handle transactions that span multiple statements?## Interview Tips* Emphasise the **try‑with‑resources** pattern – it shows you understand modern Java resource management.* Mention that in real‑world applications you rarely call `DriverManager` directly;

does	Typical usage`@RequestMapping`	Maps an HTTP request (method, path, params, headers, consumes/produces media types) to a specific handler method (or to a controller class). It can be placed at class level (defining a base URL) and at method level (defining the endpoint specifics).	`java @RequestMapping("/api/users") // class‑level base path @GetMapping("/{id}") // method‑level shortcut for GET @RequestMapping(value="/{id}", method=RequestMethod.GET) ``@RestController`	A meta‑annotation that combines `@Controller` and `@ResponseBody`. It tells Spring that the annotated class is a controller and that every handler method’s return value should be written directly to the HTTP response body (converted to JSON, XML, etc.) instead of being resolved to a view.	`java @RestController @RequestMapping("/api/books") public class BookController { @GetMapping("/{id}") public BookDto get(@PathVariable Long id) { … } } `How they work together`@RestController` marks the class as a REST controller – all methods automatically behave as if annotated with `@ResponseBody`.`@RequestMapping` (or its HTTP‑method shortcuts such as `@GetMapping`, `@PostMapping`, …) defines which URLs and HTTP verbs invoke each method.When a request matches the mapping, Spring invokes the method, serializes the returned object (via HttpMessageConverters) to JSON/XML, and writes it to the response.Advanced detailsMethod‑level attributes: `value`/`path`, `method` (GET, POST, PUT, DELETE, PATCH, OPTIONS, HEAD), `params`, `headers`, `consumes`, `produces`.Path variables and request parameters are bound with `@PathVariable` and `@RequestParam`.Content negotiation: `produces` controls the `Content‑Type` of the response; `consumes` restricts allowed request `Content‑Type`.Shortcut annotations (`@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`, `@PatchMapping`) are specialized forms of `@RequestMapping` with the HTTP method pre‑set.`@RestController` can still return `ResponseEntity<T>` to gain full control over status code and headers.Key Concepts`@RequestMapping` – URL & HTTP method routing (class‑ and method‑level).`@RestController` = `@Controller` + `@ResponseBody` (automatic serialization).Shortcut mapping annotations (`@GetMapping`, etc.).Content negotiation (`produces` / `consumes`).Integration with `@PathVariable`, `@RequestParam`, `@RequestBody`.Evaluation PointsCorrectly defines each annotation and its purpose.Shows that `@RestController` is a Injection (DI) is a design pattern that promotes loose coupling by delegating the creation and provision of an object’s dependencies to an external component (the container or framework) rather than the object creating them itself.  In Spring (and Spring Boot) the IoC (Inversion of Control) container performs the injection, allowing components to be easily tested, swapped, and maintained.Types of Dependency InjectionType	How it works	Typical Spring usage	Pros	ConsConstructor Injection	The required dependencies are supplied as arguments to the class constructor.	`@Component public class Service { private final Repo repo; public Service(Repo repo) { this.repo = repo; } }`	Immutable dependencies, enables required injection, easier to unit‑test.	More verbose if many optional dependencies.Setter Injection	Dependencies are supplied via public setter methods after the object is instantiated.	`@Component public class Service { private Repo repo; @Autowired public void setRepo(Repo repo) { this.repo = repo; } }`	Good for optional dependencies, clearer when there are many.	Allows creation of partially‑initialized objects; less immutable.Field Injection	The container injects directly into the field using reflection.	`@Component public class Service { @Autowired private Repo repo; }`	Minimal boiler‑plate, quick to write.	Harder to unit‑test (needs reflection), hides dependencies, considered less preferred by many style guides.All three approaches ultimately result in the same runtime behavior—an object receives references to its collaborators without constructing them itself. The choice depends on readability, testability, and whether the dependency is mandatory or optional.Key ConceptsInversion of Control (IoC): The framework, not the object, controls the lifecycle of dependencies.Loose Coupling: Objects depend on abstractions (interfaces) rather than concrete implementations.Spring Container: Performs DI based on configuration (annotations, XML, Java config).@Autowired / @Inject: Annotation that marks a constructor, setter, or field for injection.Required vs. Optional: Constructor injection enforces required dependencies; setter/field can be optional.Evaluation PointsProvides a clear definition of DI and its purpose (loose coupling, testability).Lists the three types with a brief description of each.Shows a concise Spring code example Spring MVCSpring MVC (Model‑View‑Controller) is Spring’s web framework built on top of the core container. It separates an application into:Model – domain objects / DTOs.View – presentation (JSP, Thymeleaf, FreeMarker, etc.).Controller – handles HTTP requests, interacts with the model, returns a view name or data.Core componentsComponent	Role`DispatcherServlet`	Front controller that receives every request and dispatches to handlers.Handler Mapping (`@RequestMapping`, `@GetMapping`, …)	Maps URLs/HTTP methods to controller methods.Handler Adapter	Invokes the matched controller method.View Resolver	Turns a logical view name into a concrete view implementation.ModelAndView	Holds model attributes and the view name returned by a controller.`@RestController`	Shortcut for REST APIs – bypasses view resolution and writes the return object directly to the response body.Typical request flowClient sends request → `DispatcherServlet`.Handler mapping finds a matching controller method.Handler adapter invokes the method.Method returns a view name (or an object for REST).View resolver renders the view → response sent back.---3. `@PostConstruct` → ready for use → optional `@PreDestroy`.Scopes – `singleton` (default), `prototype`, `request`, `session`, `application`, `websocket`.Declaration – via stereotype annotations (`@Component`, `@Service`, `@Repository`, `@Controller`) or `@Bean` methods inside a `@Configuration` class.Example```java@Servicepublic class OrderService {    private final OrderRepository repo;    public OrderService(OrderRepository repo) {   // constructor injection        this.repo = repo;    }}````OrderService` becomes a Spring bean because of the `@Service`

operator that lets you match a string column against a regular‑expression pattern.It is used for more powerful pattern matching than the simple `LIKE` wildcard operator.By default the match is case‑insensitive (unless the column uses a case‑sensitive collation or you use the `BINARY` keyword).Syntax```sqlSELECT *FROM table_nameWHERE column_name REGEXP 'pattern';````column_name` – the string column to be examined.`'pattern'` – a POSIX‑extended regular‑expression pattern.You can also write `column_name RLIKE 'pattern'` (exact same meaning).Simple Example```sql-- Find all employees whose name starts with 'J' followed by any two letters and ends with 'n'SELECT employee_id, nameFROM employeesWHERE name REGEXP '^J..n$';```Explanation of the pattern:`^`  – start of DefinitionAn Access Control List (ACL) in MySQL is the collection of privilege records (the grant tables) that define what operations a user or role may perform on which database objects (databases, tables, columns, stored routines, etc.).2. Definition of NormalizationNormalization is the systematic process of organizing data in a relational database to eliminate redundant (duplicate) data, reduce update anomalies, and ensure logical data dependencies. A column that stores comma‑separated tags must be split into separate rows or a related table.Second Normal Form (2NF)	• Table must already be in 1NF.<br>• All non‑key (dependent) columns must be fully functionally dependent on the entire primary key (not just part of a composite key).	In a table `OrderDetails(order_id, product_id, product_name, price)`, `product_name` and `price` depend only on `product_id`. Separate a table that stores both phone numbers and email addresses for a person into two one‑to‑many tables.Fifth Normal Form (5NF) (very rare)	• Table is in 4NF.<br>• No join‑dependency that is not a consequence of candidate keys.	Used in highly complex data‑warehouse schemas where a decomposition into many tables is required to avoid redundancy.Key Points1NF eliminates repeating groups → atomic columns.2NF removes partial dependencies on a composite primary key.3NF removes transitive dependencies → each non‑key attribute depends only on the primary key.Higher normal forms (BCNF, 4NF, 5NF) address more exotic anomalies but are seldom required in typical application schemas.---Evaluation PointsClear definition of normalization (avoid duplication, ensure logical dependencies).Correctly list at least the first three normal forms and describe the condition each one imposes.Mention that the first three are usually sufficient for most practical designs, with brief notes on higher forms (BCNF, 4NF, 5NF).---Common MistakesSaying 2NF only applies when the primary key is a single column (it actually concerns partial dependency on any composite key).Confusing “no repeating groups” with “no duplicate rows” (1NF is about atomicity, not row uniqueness).Claiming that 3NF eliminates all redundancy; some redundancy may remain until BCNF/4NF.---Follow‑up QuestionsHow would you convert a table that violates 2NF into 2NF? It is the source of truth for replication; slaves read the binlog and re‑execute the events to keep in sync.Traditional replication – slaves keep track of the master’s current log file name and position (`Relay_Log_Pos`). When a slave restarts it resumes from that exact location.GTID (Global Transaction ID) replication – each transaction on the master is assigned a unique identifier `<source_id>:<transaction_id>`. Slaves store the set of GTIDs they have already executed. On restart they request any missing GTIDs from the master, regardless of file/position.Advantages of GTIDFeature	Traditional (file‑position)	GTIDFailover	Requires manual `CHANGE MASTER TO MASTER_LOG_FILE='…', MASTER_LOG_POS=…` on the new master.	New master can be promoted automatically; slaves just need `START SLAVE UNTIL SQL_AFTER_GTIDS='…'`.Consistency	Risk of “hole” if a slave misses a binlog segment.	Guarantees that every transaction is applied exactly once; missing GTIDs are fetched automatically.Multi‑source	Complex to manage several masters.	Each GTID includes the source UUID, making multi‑source replication straightforward.Ease of provisioning	Must copy the correct binlog files.	New slave can be provisioned with a logical dump, then `SET GTID_PURGED=` to mark transactions as already applied.---Key ConceptsEnable GTID on both master and slaves: `gtid_mode=ON`, `enforce_gtid_consistency=TRUE`.`SHOW MASTER STATUS;` displays the current binlog file and position; `SELECT @@GLOBAL.GTID_EXECUTED;` shows the set of GTIDs the server has processed.`SQL_SLAVE_SKIP_COUNTER` works only with file‑position replication; with GTID you use `SET GLOBAL SQL_SLAVE_SKIP_COUNTER = n;` or `RESET SLAVE` + `START SLAVE UNTIL SQL_AFTER_GTIDS='…'`.---Evaluation PointsExplain that the binary log is the source of replication events.Contrast file‑position and GTID replication (how slaves track progress).Mention at least one benefit of GTID (simplified failover, automatic gap handling, multi‑source).---Common MistakesSaying the binary log is only for point‑in‑time recovery (it is also for replication).Assuming GTID removes the need for a binlog (GTID is built on top of the binlog).---Follow‑up QuestionsHow do you enable the binary log on a running MySQL instance? (`--log-bin` in my.cnf, then restart).What does the `relay_log` do on a slave?How can you safely reset GTID on a replica after a failed migration?---Interview TipsStart with: “The binary log is the write‑ahead log that records every change on the master.”Emphasize that GTID is a logical identifier that abstracts away physical log files.---Tags  MySQL, Replication, Binary Log, GTID, Master‑Slave, High Availability, HardSourceMySQL 8.0 Reference Manual – Binary Log and Replication.https://dev.mysql.com/doc/refman/8.0/en/replication.html---
